# 온콜 SOP

## 초기 단계 (사용자 < 100)

1인 운영. 대표가 사실상 24/7 1차 대응. 자동 알림 채널만 잘 갖춰서 잠 자는 동안에도 P0 는 깨우도록.

## 알림 채널

| 등급 | 채널 | 임계치 |
| --- | --- | --- |
| P0 | 카카오톡 개인 + 전화 (PagerDuty 무료 플랜) | Sentry 신규 에러 50건/5min, health 3회 연속 실패 |
| P1 | 슬랙 #alerts | Sentry 신규 에러 10건/5min, p95 > 5s |
| P2 | 이메일 일간 다이제스트 | 그 외 |

## 24시간 흐름

- **09:00** — Sentry 야간 요약 확인 (15분)
- **09:15** — 사용자 신고 큐 점검 (support@ongojisin.ai, 카톡 채널)
- **22:00** — 다음 날 배포 예정 작업 마감 (운영 변경은 가급적 새벽 X)
- **새벽** — P0 만 깨움. P1 이상은 아침 첫 일정으로 미룸.

## 일일 모니터링 체크리스트

- [ ] Sentry — 신규 에러 0건, 회귀 0건
- [ ] Fly metrics — API/AI 엔진 CPU·메모리 정상
- [ ] DB 연결 풀 사용률 < 70%
- [ ] 결제 성공률 > 99%
- [ ] AI 엔진 응답 p95 < 5s
- [ ] 한의사 가입 / 진료 기록 카운트 (Pro 전환률 추이)

## 주간 (월요일)

- [ ] 백업 정상 동작 확인 (Supabase 대시보드 backup 탭)
- [ ] 인증서 만료일 확인 (Cloudflare/Vercel 자동 갱신 동작 중인지)
- [ ] 비용 추이 점검 (OpenAI/Fly/Supabase)
- [ ] 보안 의존성 — `pnpm audit` 결과 검토

## 사용자 증가에 따른 확장

| 사용자 수 | 온콜 체계 |
| --- | --- |
| < 100 | 1인 대응 (현재) |
| 100~500 | 운영자 1명 채용 / 평일 9~22 1차 |
| 500~2000 | 24/7 로테이션 (2~3인) |
| 2000+ | PagerDuty 정식 도입, primary/secondary 분리 |

## 1인 운영의 한계 안전장치

- **자동 차단 우선** — 회복 자동화가 사람보다 빠름. 임상 안전 로직 회귀가 의심되면 Feature flag 로 즉시 차단되도록 코드에 반영.
- **백업 자동화** — Supabase 관리형 백업 활용. 직접 cron 안 굴림.
- **단일 장애점 회피** — 노트북, 휴대폰, 인터넷 회선 중 하나만 죽어도 대응 가능하도록 항상 보조 노트북 + 모바일 핫스팟 + 다른 카페 위치 확보.
